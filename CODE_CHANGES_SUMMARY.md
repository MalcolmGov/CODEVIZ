Decisions\n\n### Decision 1: API Metadata Approach\n**Question**: How to distinguish testable from non-testable APIs?\n**Decision**: Add metadata to each API object during discovery\n**Why**: \n- No database schema changes required\n- Metadata travels with API throughout system\n- UI can make decisions locally\n- Supports future extensibility\n\n---\n\n### Decision 2: Base URL Storage\n**Question**: Where to store custom base URLs?\n**Decision**: Redis (with fallback to request parameter)\n**Why**:\n- Fast lookups during test execution\n- Natural expiration (1 hour default)\n- No permanent storage needed\n- Works in distributed setup\n\n---\n\n### Decision 3: Testable Default\n**Question**: Should APIs be testable by default?\n**Decision**: Yes, mark as testable if found locally\n**Why**:\n- Local code is assumption for localhost testing\n- Reduces configuration burden\n- External APIs explicitly marked as non-testable\n- Safer default (fail if server not running)\n\n---\n\n### Decision 4: UI Separation\n**Question**: How to show different API types in UI?\n**Decision**: Separate sections with distinct visual styling\n**Why**:\n- Immediate visual clarity\n- No confusing \"grayed out\" buttons\n- Info messages explain next steps\n- Supports future interactive configuration\n\n---\n\n## Backward Compatibility\n\n✅ **Fully backward
